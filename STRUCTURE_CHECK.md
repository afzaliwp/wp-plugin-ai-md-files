# WordPress Plugin Structure & Architecture Guidelines

When generating, updating, or reviewing code for AfzaliWP plugins, you MUST strictly adhere to the following architectural and structural guidelines. This ensures compatibility with the custom autoloader, standardizes the file system, and maintains WPCS compliance.

## 1. Namespaces and Autoloading
*   **Namespace Format:** ALL classes must be within the `AfzaliWP\Plugin_Name` namespace (or its sub-namespaces). 
    *   *Example:* `namespace AfzaliWP\Cheque_Payment\Includes\API;`
*   **Class to File Mapping:** The custom autoloader converts namespaces to directory paths and class names to lowercase, hyphen-separated file names.
    *   **Rule:** Underscores `_` in class names become hyphens `-` in file names.
    *   *Example 1:* Class `WC_Hooks` in namespace `AfzaliWP\PinPoint\Includes` MUST be in the file `includes/wc-hooks.php`.
    *   *Example 2:* Class `Checkout_Editor` in namespace `AfzaliWP\PinPoint\Includes` MUST be in the file `includes/checkout-editor.php`.

## 2. Directory & File Structure
Maintain the strict, predictable folder structure shown below. Do not deviate or invent new root folders unless explicitly requested.
*   **Root Level:** Contains the main plugin file, `functions.php`, `load.php`, and documentation/package files.
*   `assets/dist/`: MUST contain compiled frontend assets (`frontend.css`, `frontend.js`, `json/`).
*   `assets/images/`: Images and icons.
*   `control-panel/`: Contains the React Application structure (if applicable), including `src`, `build`, `public`, and its own `package.json`.
*   `includes/`: Contains all PHP classes organized by sub-namespaces (e.g., `api/`).
*   `languages/`: `.po`, `.mo`, `.pot`, and WP-CLI generated fragmented `.json` files.
*   `templates/`: Frontend HTML/PHP template overrides (e.g., `checkout/slots.php`).

## 3. Main Plugin Initialization
*   **ABSPATH Check:** EVERY PHP file MUST start with `defined( 'ABSPATH' ) || die();`.
*   **Main Plugin Class:** Must be a Singleton (`get_instance()`).
    *   Hooks into `plugins_loaded` to initialize.
    *   Requires `functions.php` globally.
    *   Registers the autoloader.
    *   Defines structured constants (`PLUGIN_PREFIX_DIR`, `PLUGIN_PREFIX_URL`, etc.).
    *   Loads the text domain and conditionally enqueues scripts/styles.
    *   Calls `$this->load()` which requires `load.php`.
*   **`load.php`:** The sole purpose of this file is to instantiate the required classes (e.g., `new Settings(); new API();`). Do not place logic here.

## 4. Internationalization (i18n) & Translations
*   **Short Text Domain:** ALWAYS use the assigned short text domain for the plugin (e.g., `'pinpoint'`). Do not use long or dynamic text domains.
*   **Escaping is Mandatory:** NEVER use raw translation functions (`__()` or `_e()`) in output unless absolutely necessary and wrapped in `wp_kses()`.
    *   **Use:** `esc_html__()`, `esc_html_e()`, `esc_attr__()`, `esc_attr_e()`.
    *   *Correct:* `<p><?php esc_html_e( 'Select Time', 'pinpoint' ); ?></p>`
    *   *Incorrect:* `<p><?php _e( 'Select Time', 'pinpoint' ); ?></p>`

## 5. Coding Standards (WPCS)
*   **Formatting:** Strictly follow WordPress Coding Standards (WPCS). Use spaces after function names in declarations, spaces inside arrays, and Yoda conditions where applicable.
*   **Functions (`functions.php`):** Only for global utility functions (like `mylog`) or the autoloader itself. Wrap them in `if ( ! function_exists( 'func_name' ) )`.
*   **Classes (`includes/`):** Hook registrations (`add_action`, `add_filter`) should primarily happen inside the class `__construct()` method. Keep methods focused on a Single Responsibility.

## 6. Frontend JavaScript Architecture
*   **OOP Pattern:** Use ES6 Classes (e.g., `class PinPointDelivery { ... }`).
*   **Selectors Object:** Centralize all DOM selectors inside a class property (`selectors = { mainWrapper: '#id', ... }`). Never hardcode selector strings inside methods.
*   **State Management:** Centralize dynamic variables inside a `state` object property (`this.state = { updating: false, ... }`).
*   **Event Binding:** Keep all event listener registrations inside a dedicated `bindEvents()` method. Use delegated events `$(document).on(...)` for elements that might be re-rendered (like WooCommerce checkout updates).
*   **Data Passing:** Always use `wp_localize_script` to pass nonces, URLs, i18n strings, and settings from PHP to a global JS variable (e.g., `ppd_params`).

## 7. React / Control Panel Integration (If Applicable)
*   **Admin Enqueues:** Ensure `wpApiSettings` (containing REST URL and nonce) and `wp-i18n` are localized and passed to the React build script.
*   **Translation Merging:** Use the main plugin class to intercept and merge fragmented JSON translation files generated by WP-CLI for React components (via `pre_load_script_translations`).
